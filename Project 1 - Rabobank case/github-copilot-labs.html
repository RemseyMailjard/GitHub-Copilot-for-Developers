🧪 Lab Exercises

Scenario: The Rabobank design system team asks you to build a reusable Notification Banner UI component using plain HTML, CSS, and JavaScript. Use Copilot to scaffold the files, then verify it matches the design-system rules.
💡 Tip: Open a relevant existing HTML/CSS file in a side tab — Copilot uses open files as context for consistent style. The more examples of your design system Copilot can see, the better it aligns with Rabobank conventions.
Scenario: A legacy Rabobank internal page uses inline styles and outdated JavaScript patterns. Your team is migrating to clean external CSS and modern vanilla JS. Use Copilot to accelerate the refactoring.
💡 Tip: Copilot excels at refactoring patterns. You can use it to migrate inline styles to CSS classes, convert old-style JS to modern ES6+, or restructure HTML for better semantics. Always verify the output — subtle differences can introduce visual or behavioral bugs.
Scenario: A collega heeft een pull request ingediend met de nieuwe case-detailmodule van het Rabobank klantzaakbeheer-systeem (case-detail.js, case-card.js, formatters.js). Gebruik Copilot om een self-review te doen, documentatie te genereren en onboardingmateriaal klaar te zetten vóór de teamreview.
💡 Tip: Houd case-detail.js, case-card.js en formatters.js tegelijk open — Copilot gebruikt alle open bestanden als context. Gebruik /explain om code te begrijpen, /doc voor documentatie en @workspace voor projectbrede vragen.
Scenario: Copilot gegenereerde code voor het admin-zoekpaneel in js/components/admin.js bevat een XSS-kwetsbaarheid: gebruikersinput wordt direct via innerHTML in de DOM gezet. Zoekresultaten tonen daardoor klantnamen en zaaknummers ongesanitized. Identificeer de kwetsbaarheid, fix hem en leer hoe je Copilot-generated code kritisch beoordeelt in een bancaire context.
⚠️ Key lesson: Copilot begrijpt jouw beveiligingscontext niet. Gegenereerde code die gebruikersinput in de DOM plaatst moet altijd worden beoordeeld op XSS-risico. In een bancaire omgeving is élke gebruikersinput potentieel kwaadaardig — gebruik nooit innerHTML met ongesanitized data.
Scenario: Your team is building a mortgage calculator for the Rabobank customer portal. You need comprehensive test coverage before release. Use Copilot to generate and improve tests for plain JavaScript functions.
💡 Tip: Copilot generates better tests when the function under test is well-named and documented. Keep your source file and test file open side by side — Copilot uses both as context. Use /tests for quick generation, then refine with specific prompts for edge cases.
Scenario: The Rabobank customer account dashboard loads slowly because of inefficient DOM manipulation and data handling in plain JavaScript. Use Copilot to identify performance issues and improve code quality.
💡 Tip: When asking Copilot to optimize, be specific about the constraint: number of records, response time targets, or bundle size limits. Generic "make it faster" prompts give generic answers. Rabobank-scale data (millions of accounts) needs different optimizations than a small app.
Scenario: You need to build a REST endpoint for an internal Rabobank service that retrieves customer notification preferences. Use Copilot to scaffold the controller, service, and DTO layers following Rabobank coding standards.
💡 Tip: Keep a CODING_STANDARDS.md file open in your project — Copilot uses it as context to align generated code with your conventions.
Scenario: Your team has specific Rabobank coding conventions (naming, error handling, logging format). Explore how providing Copilot with context files changes the quality of suggestions.
💡 Tip: The .github/copilot-instructions.md file is automatically used by Copilot as context for all suggestions in the repository. This is the most impactful way to enforce team-wide coding standards. Share it via your Git repo so the whole Rabobank team benefits.
Scenario: A new team member is onboarding and needs to understand a complex Rabobank payment processing microservice. Use Copilot to generate architecture documentation, code explanations, and a system overview from the existing code.
💡 Tip: Use @workspace for project-wide context, /explain for detailed method walkthroughs, and /doc for generating documentation. For API specs, keep your controller and DTO files open side by side.
Scenario: Copilot generated a database query method that concatenates user input directly into SQL. Find the vulnerability and fix it using parameterized queries.
⚠️ Key lesson: In a banking context, every generated query must be reviewed for injection risks. Never concatenate user input into SQL strings. Always mask PII (BSN, IBAN, passwords) in logs. Copilot generates code that works, but security requires human judgment.
Scenario: You're developing a BankAccountService for Rabobank's core banking platform. The service handles deposits, withdrawals, and transfers. You need comprehensive test coverage before the code can pass the Rabobank quality gate.
💡 Tip: The most valuable tests aren't the ones Copilot generates automatically — they're the edge cases you think of based on your domain knowledge. Use Copilot to write the boilerplate, then add banking-specific scenarios manually. A good prompt: "What edge cases would a financial auditor test?"
Scenario: A Rabobank batch processing service generates monthly account statements but has performance problems at scale and code quality issues flagged in the last SonarQube scan. Use Copilot to systematically improve the code.
💡 Tip: Tackle code quality improvements one category at a time (readability → performance → reliability → security). Copilot gives better results with focused prompts than "fix everything." For Rabobank-scale applications, always ask about the data volume implications.
